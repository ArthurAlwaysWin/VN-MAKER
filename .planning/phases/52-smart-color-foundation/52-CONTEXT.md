<decisions>
## Implementation Decisions

### Module Structure
- **D-01:** Create new `src/engine/oklch.js` module (do NOT extend `colorHarmony.js` — it remains for backward compat with PaletteModal)
- **D-02:** Export 3 main functions: `rgbToOklch(r, g, b)`, `oklchToRgb(l, c, h)`, `deriveTokens(primaryHex, accentHex, mode)`
- **D-03:** Rule table is inline in `oklch.js` (not a separate data file) — rules are tightly coupled to derivation logic, <100 entries

### Color Math
- **D-04:** OKLCH color space via pure JS implementation (~60 lines of matrix math), zero external dependencies
- **D-05:** Gamut clamping uses binary-search chroma reduction (bisect C until sRGB-valid, preserving L and H)
- **D-06:** Output format is `rgba()` strings matching existing DEFAULT_TOKENS format — downstream consumers don't need to know about OKLCH

### Derivation Rules
- **D-07:** 5 derivation groups per research: primary-alpha (7 tokens), accent-alpha (2), text-opacity (13), background (6), special (7)
- **D-08:** `danger` and `danger-hover` are fixed constants (UX convention, not derived)
- **D-09:** Gradient tokens (`dialogue-bg`, `title-bg`) use template strings with derived base colors injected
- **D-10:** Dark/light mode handled via LIGHT_OVERRIDES table that selectively overrides DARK_RULES (dark is the default)

### API Contract
- **D-11:** `deriveTokens(primaryHex, accentHex, mode)` returns `Record<string, string>` with all 35 color token keys (excludes font, radius, blur tokens)
- **D-12:** When `accentHex` is null/undefined, auto-derive accent via complementary harmony (hue + 180°)
- **D-13:** Return value is a plain object ready for `setTokenBatch()` — no intermediate format needed

### Agent's Discretion
- Internal OKLCH math implementation details (matrix coefficients, cube root functions)
- Exact binary-search tolerance for gamut clamping
- Test structure and test helper utilities
