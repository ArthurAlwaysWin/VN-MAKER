# Focused Research: v1.6 完整主题包与图片主题产出（FEATURES / UX）

**Project:** Galgame Maker  
**Milestone:** v1.6 完整主题包与图片主题产出  
**Dimension:** Features / User Experience  
**Researched:** 2026-04-27  
**Confidence:** HIGH for current-product gaps and milestone fit; MEDIUM for broader “成熟 VN 主题体验” baseline

---

## 结论先行

v1.6 的用户感知目标不应是“现在可以导入一些主题相关文件”，而必须是：

> **“我拿到一个 `.gmtheme` 文件，看到清楚的风格预览，点一次导入/应用，就得到一套完整、可复用、不会缺图的成品主题。”**

当前仓库已经具备图片主题基础能力，但**用户体验仍停留在“散落槽位 + 轻量预设”**：

- `ThemePackageModal.vue` 只有文字卡片 + 色条，没有真正的视觉预览、覆盖范围说明、完成度说明
- `themePackager.js` 的 `formatVersion: 1` 仍只覆盖 `tokens + nineSlice + preview.png`，离“完整主题包”不够
- `builtinThemes.js` 中多套主题仍以**颜色差异为主**，不是“完整图片主题”

因此，v1.6 的需求切线应该是：**把主题从“配置集合”升级为“可浏览、可导入、可导出、可复用的成品包”。**

## 1) 用户对“完整主题包导入 / 导出工作流”的预期

## 用户真正期望的不是“能打包”，而是“能无脑复用”

| 用户预期 | 为什么是刚需 | 对 v1.6 的 requirement implication |
|---|---|---|
| **一个文件完成分享** | 主题分享的心智模型就是“发一个包给别人” | `.gmtheme` 必须是单文件，包含配置 + 全部引用图片资产，导入后不可再手动补路径 |
| **导入前知道自己会得到什么** | 用户不想先应用再发现只是换了几个颜色 | 导入器/选择器必须展示主题封面、关键预览、覆盖范围、作者/版本/兼容性 |
| **导入后立即可用** | “一键主题”不接受逐槽位补图 | 导入成功后必须自动落地到项目资源并一次性连通全部槽位 |
| **不会污染或打坏现有项目** | 主题切换是高频试错行为 | 导入必须非破坏式：预览/确认/应用三段式，且应用支持单步撤销 |
| **导出内容可被别的项目稳定导入** | 用户会把主题当资产复用 | 导出包必须带 `formatVersion`、兼容信息、manifest、预览图、coverage 信息 |
| **缺项时要明确提示，不要默默降级** | 否则“完整主题”承诺落空 | 导入器必须区分：完整主题 / 部分主题 / 旧格式兼容导入 |

### requirement-level implications

- **PKG-UX-01**: `.gmtheme` 导出必须打包**完整主题所需全部资源**，不能只导出颜色或部分图片引用。
- **PKG-UX-02**: `.gmtheme` manifest 必须至少包含：`id / name / description / author / version / formatVersion / previewImages / coverage / tags / compatibility`。
- **PKG-UX-03**: 导入流程必须先显示“导入摘要”，再允许应用；摘要至少含主题名、作者、预览、覆盖范围、兼容/升级提示。
- **PKG-UX-04**: 导入后必须自动把资源复制到项目内受控目录，并完成路径重写；用户不应看到裸文件路径修复步骤。
- **PKG-UX-05**: 对旧 `.theme` / 旧 `.gmtheme` 必须以“兼容导入（部分覆盖）”方式明确标识，而不是假装它是完整主题。
- **PKG-UX-06**: 应用主题必须是**单事务**，支持一次 `Ctrl+Z` 撤销整个应用。

### 不应接受的弱体验

- 只显示“导入成功”，但用户不知道哪些槽位没覆盖
- 导入后还要去 10+ 个图片槽手动补图
- 主题包里有图，但选择器看不到实机预览
- 导出后的主题跨项目导入时出现丢图/错路径

## 2) 好的主题浏览 / 选择 / 预览流程应包含什么

## 推荐流程：浏览列表 → 选中卡片 → 看多场景预览 → 看覆盖说明 → 应用

当前 `ThemePackageModal` 只够做“预设列表”，不够做“主题浏览器”。  
v1.6 应升级为真正的 **Theme Browser**。

### 主题浏览器最低配要素

| 模块 | 必须包含 | 为什么 |
|---|---|---|
| **卡片列表** | 封面图、主题名、风格一句话、标签、来源（内置/导入） | 用户先按视觉风格筛，不是按 JSON 名称筛 |
| **完成度标识** | `完整主题` / `部分主题` / `旧格式` 徽章 | 防止“看起来很多，实际只换颜色” |
| **覆盖范围标识** | 对话框、按钮、四大界面、图标/光标、控件风格等 badges | 用户必须快速判断是否是全套 |
| **预览区** | 至少 3-4 张固定视角预览：对话框、Save/Load、Settings、QAB/按钮 | 单一封面不足以判断主题质量 |
| **应用状态** | 已应用 / 可更新 / 兼容导入 | 帮助用户管理已导入主题 |
| **操作** | 预览、应用、导入、导出、删除（仅导入主题）、复制/另存 | 主题是资产，不只是按钮 |

### 预览流必须是“固定视角 + 代表性界面”

推荐固定四视角：

1. **Dialogue 视角**：主对话框 + 名牌 + 继续指示
2. **Save/Load 视角**：背景、slot、分页/页签
3. **Settings 视角**：tab、slider、toggle、panel
4. **Action/Button 视角**：QAB、关闭按钮、菜单按钮、图标/光标

这四个视角能覆盖用户对“这是不是真正完整主题”的 90% 判断。

### requirement-level implications

- **BRW-01**: 主题选择器必须从“文字卡片”升级为“图像卡片 + 预览面板”。
- **BRW-02**: 每个主题必须提供至少 1 张 hero 图 + 3 张固定模块预览图；选择器不可只显示色条。
- **BRW-03**: 选择器必须显示 coverage badges：`对话框 / 按钮 / SaveLoad / Backlog / GameMenu / Settings / 图标 / 光标`。
- **BRW-04**: 选择器必须区分 `内置`、`导入`、`当前已应用` 三种状态。
- **BRW-05**: 应用前支持“先看后用”；不要求复杂编辑，但必须能看完整预览。
- **BRW-06**: 预览图必须来自真实主题数据渲染结果，而不是纯手工宣传图；否则不可信。

### 明确不建议

- 不要做只显示缩略色块的卡片
- 不要把主题详情藏进二级弹窗后才看得到
- 不要让用户靠主题名猜风格差异
- 不要让“完整主题”和“只有配色的旧主题”混在一层没有区分

## 3) 在本产品里，什么才算“完整主题”

## 定义：完整主题 = 视觉覆盖完整 + 风格语言一致 + 自解释 + 可预览

对 Galgame Maker 来说，“完整主题”不能再是“有一些 tokens + 几张图片”。  
它必须达到**可交付成品 UI 皮肤**的程度。

### 完整主题最低覆盖面

| 类别 | 必须覆盖 | 说明 |
|---|---|---|
| **对话框** | 主框图、名牌图、装饰层、文字安全区结果 | 对话框是最核心感知面 |
| **按钮族** | `game-menu-button`、`QAB`、`close-button family`、`page-tab/pager`、`settings-tab` 全状态 | v1.5 已打通槽位，v1.6 必须把这些真正填满 |
| **四大界面** | SaveLoad、Backlog、GameMenu、Settings 的背景与装饰完整 | 否则仍是“局部主题” |
| **控件风格** | tab/toggle/slider/panel/button 的 style 与主题气质一致 | 不能图片很日式、控件却还是默认软件感 |
| **图标 / 光标** | 至少核心 action icon slots + default/pointer cursor | 小元素决定成品感 |
| **元数据** | 名称、作者、描述、版本、标签、预览图、兼容范围 | 主题要能被浏览、管理、分享 |

### 完整主题一致性标准

不是“把所有槽都塞图”就算完成，还必须满足：

- **形状语言一致**：例如和风主题用卷轴/缎带/木框，不应混入现代直角蓝色 tab
- **材质一致**：纸感、玻璃感、金属感、布料感至少要前后一致
- **密度一致**：装饰繁复主题不能只有对话框华丽、设置页却空白
- **可读性一致**：亮/暗、对比度、按钮文字可见性不能在不同界面大幅波动

### 完整主题的包内元数据要求

| 字段 | 用途 | v1.6 必须性 |
|---|---|---|
| `id` | 稳定识别主题 | 必须 |
| `name` / `description` | 浏览器展示 | 必须 |
| `author` / `version` | 分享与更新判断 | 必须 |
| `formatVersion` | 升级兼容 | 必须 |
| `compatibility` | 标示最低/目标 app 版本 | 必须 |
| `tags` | 风格筛选（和风/现代/浅色/深色/奇幻等） | 必须 |
| `previewImages` | 主题浏览器预览 | 必须 |
| `coverage` | 标示覆盖到哪些模块 | 必须 |
| `tone` / `lightDark` | 帮助快速匹配题材 | 应该有 |

### requirement-level implications

- **CMP-01**: v1.6 若对外称“完整主题”，则 5 套内置主题必须全部达到相同 coverage bar，不能出现 1 套是全图、4 套只是改色。
- **CMP-02**: 主题包 schema 必须显式记录 coverage，而不是运行时猜测。
- **CMP-03**: 主题浏览器必须向用户显示“完整度”，而不是默认所有主题都一样完整。
- **CMP-04**: 每套完整主题都必须附带预览图集；没有预览图的主题不应进入“完整主题”列表。

## 4) 5 套 shipped themes 应该如何区分、如何呈现

## 原则：5 套不是 5 个色板，而是 5 个可立即套用的题材入口

建议 v1.6 的 5 套完整图片主题覆盖最常见 VN 题材：

| 主题方向 | 推荐定位 | 必须做出的差异 | 展示文案重点 |
|---|---|---|---|
| **和风古典** | 古风、和风、神社、历史恋爱 | 宣纸/木纹/缎带 tab/印章感 icon | “温暖、传统、装饰感强” |
| **现代都市** | 校园、职场、都市恋爱 | 玻璃/亚克力/细线框/清爽蓝白 | “清晰、轻盈、商业感 UI” |
| **暗黑幻想** | 奇幻、冒险、魔法、悬疑 | 金属边框/宝石高光/厚重暗色/华丽装饰 | “厚重、戏剧化、强存在感” |
| **日常浅色** | 轻松、校园、治愈、女性向 | 浅底、圆角、低压迫、柔和图标 | “明亮、亲和、适合日常向” |
| **科幻霓虹** | 未来、赛博、悬疑、机能感 | 梯形/切角/tab 光边/高对比图标 | “锐利、科技、强识别度” |

### 这 5 套必须在以下维度至少拉开 3 项以上

- **明暗极性**：至少 2 套浅色，3 套深色，避免全是暗主题
- **控件轮廓**：圆角 / 直角 / 梯形 / 缎带至少形成明显差异
- **装饰密度**：极简 vs 中等 vs 华丽要都有
- **材质语言**：纸、玻璃、金属、软色卡片、霓虹面板
- **图标 / 光标气质**：不能全复用同一套图标仅换颜色

### 主题卡片展示要求

每套 shipped theme 的卡片至少要展示：

- **Hero 图**
- **3 个风格标签**（如 `和风` / `深色` / `装饰型`）
- **适用题材一句话**（例如“适合古风、神怪、和风恋爱作”）
- **覆盖范围徽章**
- **Light / Dark 标识**
- **“完整主题”徽章**

### requirement-level implications

- **SHIP-01**: 5 套内置主题必须全部是**完整图片主题**，不得包含“只有 tokens 改色”的占位主题。
- **SHIP-02**: 5 套主题的区分依据必须包含**形状语言与材质语言**，不允许只换色相。
- **SHIP-03**: 主题列表必须按“题材/风格”展示，不应只按内部 ID 展示。
- **SHIP-04**: 每套主题都要附带“推荐题材”描述，帮助不会设计的用户选型。
- **SHIP-05**: 默认/旧式基础主题应与 v1.6 的 5 套完整主题视觉分层展示，避免误导用户把“基础默认”当成同等级成品主题。

## 5) 对 roadmap 最有用的需求归纳（建议直接进入 v1.6 requirement pool）

## A. 主题包工作流

- **REQ-PKG-01**: 用户可将当前完整图片主题导出为单个 `.gmtheme` 文件，文件内包含完整配置、预览图与全部引用 UI 资产。
- **REQ-PKG-02**: 用户导入 `.gmtheme` 时，系统先展示主题摘要（名称、作者、版本、预览、覆盖范围、兼容提示），确认后再应用。
- **REQ-PKG-03**: 导入后的主题会自动复制到项目受控目录并完成路径重写，无需用户手动补资源。
- **REQ-PKG-04**: 旧格式主题可被兼容导入，但必须标记为“部分主题”或“旧格式升级导入”。
- **REQ-PKG-05**: 主题应用支持单步撤销，且导入失败不会留下半应用状态。

## B. 主题浏览器 / 选择器

- **REQ-BRW-01**: 主题选择器支持内置主题与导入主题统一浏览，但来源状态必须可见。
- **REQ-BRW-02**: 每个主题卡片至少显示 hero 图、名称、风格标签、完整度、覆盖范围。
- **REQ-BRW-03**: 选中主题后可查看固定四视角预览：Dialogue / SaveLoad / Settings / Action。
- **REQ-BRW-04**: 用户可直接从浏览器执行应用、导入、导出；不应在多个分散入口跳转。

## C. 完整主题定义

- **REQ-CMP-01**: v1.6 的“完整主题”必须覆盖对话框、五个按钮族、四大界面、核心图标、光标、控件风格。
- **REQ-CMP-02**: 主题 manifest 必须携带 coverage 信息，供浏览器和导入器展示。
- **REQ-CMP-03**: 没有预览图集或 coverage 未达标的主题，不得标记为“完整主题”。

## D. 5 套 shipped themes

- **REQ-SHIP-01**: v1.6 内置 5 套完整图片主题，且全部具备可导出、可导入、可复用的一致包结构。
- **REQ-SHIP-02**: 5 套主题必须按题材和风格拉开显著差异，至少覆盖：和风、现代、幻想、浅色日常、科幻/高识别度。
- **REQ-SHIP-03**: 每套主题都附带风格标签、推荐题材、hero 图、固定预览图集。

## 6) 明确的非目标（防止 milestone 又散回“图片槽位补丁”）

| 非目标 | 为什么这次不该做 |
|---|---|
| 主题市场 / 在线下载中心 | 先把本地主题包体验做完整 |
| 主题编辑器自由画布化 | 会把主题成品交付问题拖成新编辑器系统 |
| 每场景 / 每页面主题覆盖 | 与“完整、可复用、一键应用”主目标相冲突 |
| 社区评分 / 收藏 / 云同步 | 都是二级生态能力，不是 v1.6 核心 |

---

## 7) 里程碑验收时最该问的 5 个问题

1. **用户拿到一个 `.gmtheme`，是否能在导入前看懂它是什么风格、覆盖了哪些部分？**
2. **导入后是否真的“一键可用”，而不是继续补图补路径？**
3. **选择器是否能让用户一眼区分 5 套主题，而不是看起来像 5 个换色版本？**
4. **5 套 shipped themes 是否全部达到同一“完整主题”标准？**
5. **旧格式主题是否被诚实地标注为兼容导入，而不是伪装成完整主题？**

只要这 5 个问题里有 2 个答不上来，v1.6 就仍然会被感知为“图片槽位继续堆功能”，而不是“完整主题包完成”。
