# Domain Pitfalls — v1.6 完整主题包与图片主题产出

**Domain:** 完整主题包 / 图片主题导入导出 / 5 套内置完整主题  
**Researched:** 2026-04-27  
**Confidence:** HIGH — based on direct repo inspection of current v0.6-v1.5 theme/package/image pipeline

## Executive Priority

**Most likely milestone killers:**

1. **“格式升级成功”但实际是静默丢字段**  
   当前 `.theme` 只覆盖 `tokens + nineSlice`，如果 v1.6 在旧导入链路上硬塞完整主题，最容易出现“导入成功，但按钮/界面/图标/对话框没跟上”。
2. **资产导入语义不清，重复导入后项目越用越胖**  
   当前资产导入默认 `uniqueFilename()` 自动改名；若完整主题包也沿用这套语义，重复导入同一主题会不断生成 `-1/-2/-3` 文件，引用和清理都会失控。
3. **预览链路和运行时链路不是同一条路**  
   现在 preview 主要靠 `update-theme` / `update-screen-layout` 分开发消息；完整主题包是跨 `ui.theme + ui.widgetStyles + ui.<screen> + dialogue/icon/cursor/buttonFamily` 的整体包，如果不改成同一份 package snapshot 驱动，预览必漂移。
4. **5 套主题内容生产过早并行**  
   如果在 package contract、导入重映射、预览 parity 没冻结前就同时产 5 套图，后面格式一改就是 5 倍返工。

## Critical Pitfalls

### Pitfall 1: 导入/导出“成功”，但资产引用在落地后断掉

**Requirement defenses:**

1. v1.6 新包格式必须显式区分“包内路径”与“项目路径”
2. 导入完成条件不是“parse 成功”，而是“所有引用已落地并重映射成功”
3. 导入后必须跑一次 reference audit
4. 导出必须只导 canonical refs，不导任意绝对路径/临时路径

**Roadmap implication:** 先做 import remap contract，再做批量主题内容。

### Pitfall 2: 重复资产膨胀、覆盖语义不清、文件碰撞后不可控

**Requirement defenses:**

1. 完整主题包不得导入到裸 `assets/ui/` 根目录
2. 必须命名空间化：`assets/ui/themes/<packageId>/...`
3. 必须定义重导语义
4. 必须有 manifest
5. 删除主题时按 manifest 清理，只删该包拥有的文件

**Roadmap implication:** “包命名空间 + manifest + 重导策略”必须进第一阶段。

### Pitfall 3: 包格式兼容/版本漂移导致“看似兼容，实际降级”

**Requirement defenses:**

1. v1.6 必须正式切到新扩展名 / 新 package type：`.gmtheme`
2. 旧 `.theme` 只作为 legacy import
3. 新包必须 fail closed
4. 必须有 migration layer
5. 导入结果页必须告诉用户：导入了哪些范围，跳过了哪些范围

### Pitfall 4: 主题卡片预览和实际应用结果不一致

**Requirement defenses:**

1. 完整主题预览必须走“整包 snapshot”
2. 预览和应用必须共用同一 normalization pipeline
3. 验收必须覆盖至少 3 个视图
4. 卡片缩略图只能做 marketing，不可当验收依据
5. 必须有 packaged-theme parity tests

**Roadmap implication:** 先做 package preview protocol，再升级 ThemePackageModal。

### Pitfall 5: 主题选择 / 应用流程让用户误覆盖现有工作

**Requirement defenses:**

1. Apply 前必须展示覆盖范围
2. 必须区分两种动作：预览 / 应用
3. 必须给出 overwrite 语义文案
4. 必须把 undo 当 safety net，不是 primary explanation
5. 导入外部包时，必须先做 preflight，再允许 apply

### Pitfall 6: 一口气交付 5 套完整主题，内容生产把工具链问题放大 5 倍

**Requirement defenses:**

1. 先做 1 套 golden theme，再批量复制
2. 先冻结资产清单 matrix
3. 每套主题必须过同一 acceptance checklist
4. 5 套内容生产应放在 package pipeline 稳定之后
5. 如里程碑压力大，优先保“3 套完整 + 2 套延期”，不要交 5 套半成品

**Roadmap implication:** 最安全顺序是：

1. 包格式 / 兼容契约
2. 导入落地与碰撞策略
3. 预览 / 应用 parity
4. 1 套 golden theme
5. 其余 4 套批量产出

## Hard Requirements This Milestone Should Add

1. 完整主题包必须覆盖完整 `ui` snapshot，而不是只有 `ui.theme`
2. 旧 `.theme` 明确作为 legacy import，不承诺完整主题能力
3. 包导入资产必须进入包命名空间，而不是裸 `assets/ui/`
4. 导入成功 = 所有 package refs 已 remap 为 canonical `ui/...` 且文件存在
5. 完整主题包预览、实际应用、导出结果必须一致
6. 用户在 apply 前必须知道会覆盖哪些范围
7. 在第一套完整主题跑通前，不开始大规模 5 套并行生产

## Recommendation

不要把 v1.6 当成“做一个更大的 ThemePackageModal”。  
它本质上是一个 **package contract + asset ownership + preview parity + content factory** 里程碑。
