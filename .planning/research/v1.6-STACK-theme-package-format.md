# v1.6 STACK Research — Theme Package Format (`.gmtheme`)

**Project:** Galgame Maker  
**Milestone:** v1.6 完整主题包与图片主题产出  
**Dimension:** STACK / PACKAGE FORMAT  
**Researched:** 2026-04-27  
**Confidence:** HIGH for repo findings, MEDIUM for mature-tooling comparison

## Executive Recommendation

Use **`.gmtheme` = ZIP + `theme.json` + `assets/`**, but make it a **project-path-native package**, not a special image container.

**Recommended core rule:** all runtime image references inside `theme.json` must use the project's existing canonical asset form, i.e. **`ui/...` relative to project `assets/`**. Inside the zip, those files live at **`assets/ui/...`**. This matches the current runtime/export pipeline and avoids adding a second path model.

**Strong recommendation:** namespace every imported theme under:

```text
assets/ui/themes/<themeId>/...
```

So manifest paths become:

```json
"ui/themes/wafuu-sakura/dialogue/panel.png"
```

This prevents collisions, makes ownership clear, and makes import/export deterministic.

## Why this is the right fit for this repo

Current code already assumes:

- canonical UI asset paths are `ui/...` (`src/shared/uiImageContract.js`)
- runtime transport paths like `asset://ui/...` and `/game/ui/...` are only runtime wrappers, not source-of-truth
- export scans `script.json` for `ui/...` references and copies from `project/assets/...` (`src/engine/scanAssets.js`, `electron/exportGame.js`)
- current `.theme` support is too small: it only handles `ui.theme.tokens` + some `nineSlice`, not full bundles (`src/utils/themePackager.js`, `PresetModal.vue`)

So v1.6 should **upgrade the package format to match the existing project asset model**, not invent a parallel one.

## Recommended `.gmtheme` Package Structure

### Zip layout

```text
my-theme.gmtheme
├── theme.json
└── assets/
    ├── _meta/
    │   └── preview.png
    └── ui/
        └── themes/
            └── <themeId>/
                ├── dialogue/
                ├── buttons/
                ├── chrome/
                ├── cursors/
                ├── icons/
                └── screens/
```

### Path rule

- **Inside `theme.json` UI config:** use canonical project-style paths only:
  - `ui/themes/<themeId>/...`
- **Inside ZIP:** actual file location is:
  - `assets/ui/themes/<themeId>/...`

### Manifest shape

```json
{
  "formatVersion": 2,
```

## Scope Recommendation: what the package should own

A `.gmtheme` should own only **reusable UI-theme data**, not game-specific content.

### Include

- `ui.theme`
  - `tokens`
  - `colorRecipe`
  - `nineSlice`
  - `buttonFamilies`
  - `cursor`
  - `icons`
- `ui.widgetStyles`
- `ui.dialogueBox`
- `ui.saveLoadScreen`
- `ui.backlogScreen`
- `ui.gameMenu`
- `ui.settingsScreen`

### Exclude

- story/scenes
- characters/backgrounds/audio
- arbitrary title page composition if it is project-specific page content

**Practical call:** do **not** package arbitrary `titleScreen.elements` unless v1.6 explicitly defines a reusable title-screen chrome subset. Otherwise theme import will overwrite game-specific title design.

## Asset Ownership Rules

### Rule 1: package-owned assets live only under its namespace

Each imported package owns only:

```text
assets/ui/themes/<themeId>/**
```

Never scatter imported theme files into flat `assets/ui/`.

### Rule 2: manifest references must stay stable

Do **not** use the repo's current `uniqueFilename()` auto-rename strategy for theme package import.  
That logic is fine for ad hoc user asset import, but wrong for package import because it breaks manifest references.

For `.gmtheme` import, paths must be deterministic.

### Rule 3: copy/overwrite behavior

On import:

- if destination file missing → copy
- if destination exists and hash matches → skip copy
- if destination exists, hash differs, and path is inside the same package namespace → overwrite
- if path escapes namespace or uses invalid path form → reject import

### Rule 4: no auto-delete on apply

Applying a theme should **copy/update**, not garbage-collect old theme files.  
Deletion can be a later cleanup tool. This avoids breaking undo, mixed themes, or old references.

## Import / Export Requirements

### Import

1. unzip `.gmtheme`
2. validate `theme.json`
3. reject path traversal (`..`, absolute paths, backslashes, drive letters)
4. validate all runtime asset refs are canonical `ui/...`
5. validate every referenced file exists in zip under `assets/...`
6. validate file types with existing UI image validator (`png/jpg/jpeg/webp`)
7. copy assets into project
8. atomically apply UI subtrees to `script.json`
9. save package metadata into script for round-trip/debugging

Recommended metadata written into project:

```json
"ui": {
  "theme": {
    "packageMeta": {
      "id": "wafuu-sakura",
      "packageVersion": "1.0.0",
      "formatVersion": 2,
      "assetRoot": "ui/themes/wafuu-sakura/"
    }
  }
}
```

### Export

When exporting current project theme as `.gmtheme`:

1. collect all theme-owned UI config sections
2. scan referenced UI assets
3. if referenced files are outside `ui/themes/<themeId>/`, copy them into that namespace in the package and rewrite manifest paths
4. emit a self-contained zip
5. include preview image
6. include file hashes for deterministic re-import

**Important:** export should produce a **portable snapshot**, not depend on original project folder layout beyond canonical `ui/...`.

## Versioning and Migration

### Use 2 separate versions

#### 1. `formatVersion` (schema)

Use integer:

- `1` = legacy `.theme`
- `2` = full `.gmtheme`

#### 2. `packageVersion` (content revision)

Use semver:

- `1.0.0`
- `1.1.0`
- etc.

This cleanly separates schema migration from theme-art updates.

### Migration policy

#### Keep legacy import

v1.6 should still import old `.theme` files as a migration path.

#### Stop legacy export

After v1.6, export only `.gmtheme`.

#### Legacy conversion behavior

Old `.theme` imports should be normalized into v1.6 shape:

- convert embedded image refs into real files under `ui/themes/<imported-id>/legacy/...`
- rewrite any surviving data URLs to canonical `ui/...` refs
- fill missing sections with empty objects

## Compatibility Pitfalls

### 1. `data:` images are a trap

Current code still tolerates some legacy/data-url cases, but v1.6 packages should **not** ship runtime images as `data:` URIs.

### 2. `asset://` must never be stored in package manifests

`asset://` is runtime transport, not source format.

Store:

```text
ui/themes/foo/bar.png
```

Not:

```text
asset://ui/themes/foo/bar.png
/game/ui/themes/foo/bar.png
assets/ui/themes/foo/bar.png
C:\...
```

### 3. Partial apply causes broken hybrid themes

Current `.theme` import only updates `ui.theme`.  
For v1.6, applying a package must update the whole themeable bundle atomically, or users get mismatched tokens/images/layouts.

### 4. Flat `assets/ui/` will become unmanageable

Without namespacing, 5 complete image themes will collide fast (`panel.png`, `button.png`, `hover.png`, etc.).

### 5. Existing ad hoc asset import rules are not enough

Current asset import uses filename conflict suffixing. Good for manual import, bad for package manifests.

## Mature Tooling Comparison

Mature VN tooling expects UI themes to be **asset-first**, not token-first.

Official Ren'Py docs still model GUI assets as named files like:

- `gui/textbox.png`
- `gui/game_menu.png`
- `gui/button/choice_idle_background.png`
- `hover_background`
- `selected_hover_background`

That matches the direction of v1.6: **package a coherent set of referenced UI files plus a manifest**, not just a color preset.

## Milestone Requirements I would lock

1. `.gmtheme` is ZIP-based and replaces `.theme` for export
2. `theme.json` stores canonical project-style `ui/...` paths only
3. package assets mirror project layout under `assets/`
4. theme assets are namespaced under `ui/themes/<themeId>/`
5. import validates paths, formats, and compatibility before writing
6. import uses deterministic overwrite/skip rules, not auto-renaming
7. export produces a self-contained portable bundle
8. legacy `.theme` import remains supported as migration
9. package metadata is stored in `script.json`, not a second project file

## Bottom-line Recommendation

For this codebase, the best v1.6 format is:

**`.gmtheme` = self-contained ZIP snapshot of reusable UI config + namespaced UI assets, using existing canonical `ui/...` project paths.**

That is the smallest-change path that:

- fits current runtime/export architecture
- supports 5 complete image themes cleanly
- avoids path chaos
- gives future migration room
