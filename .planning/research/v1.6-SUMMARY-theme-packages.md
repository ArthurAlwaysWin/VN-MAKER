# v1.6 综合规划摘要 — 完整主题包与图片主题产出

**Milestone:** v1.6 完整主题包与图片主题产出  
**用途:** 给下一步 requirements / roadmap 直接使用，不做泛泛研究复述  
**结论基线:** 先把 `.gmtheme` 做成**可安装、可预览、可导出、可复用**的完整主题包，再按 **1 套 golden theme → 其余 4 套复制扩展** 的顺序交付内容。

---

## 1. 推荐 scope（建议直接锁进 milestone）

v1.6 应被定义为一个 **package contract + install/apply workflow + shipped content** 里程碑，而不是“继续给主题系统加几个图片槽位”。

### In scope

1. **`.gmtheme` 完整包**
   - 包含主题配置 + 预览元数据 + 全部引用 UI 图片资产
   - 导入后自动安装到项目内，不要求用户手动补路径

2. **完整工作流**
   - 浏览 / 预览 / 导入 / 安装 / 应用 / 导出
   - 内置主题与导入主题走同一条 apply 路径

3. **完整主题标准**
   - 至少覆盖：`ui.theme`、`ui.widgetStyles`、`ui.dialogueBox`、`SaveLoad`、`Backlog`、`GameMenu`、`Settings`、核心 icons、cursor、button families

4. **5 套完整图片主题**
   - 交付目标仍是 5 套
   - 但 requirements 与 roadmap 必须明确：**先做 1 套 golden theme 跑通验收，再扩展剩余 4 套**

### Out of scope

- 在线主题市场 / 云同步
- 每页面 / 每场景局部主题覆盖
- 字体打包入主题（体积与许可证风险高）
- 把主题系统扩展成自由画布式“UI 设计器”

---

## 2. 必须锁死的 hard requirements

1. **运行时唯一真源仍是 `script.json`**
   - 不引入“运行时从 theme registry 二次 overlay”新机制

2. **项目内 canonical 路径不变**
   - 主题配置里图片引用只允许 `ui/...`
   - 禁止写入 `asset://`、绝对路径、`assets/...`、data URI

3. **所有完整主题资产必须安装到命名空间**
   - `assets/ui/themes/<themeId>/**`
   - 禁止扔到裸 `assets/ui/`

4. **导入成功的定义必须提高**
   - 不是 parse 成功
   - 而是：所有引用已校验、已落盘、已 remap 为 canonical `ui/...`、应用后无缺图

5. **apply 必须是整包原子操作**
   - 一次性替换完整 theme bundle 负责的 UI 区块
   - 支持单步 undo
   - 禁止只改 `ui.theme` 造成 hybrid theme

6. **导入前必须能看懂主题**
   - 至少展示：名称、作者、版本、预览图、coverage、兼容信息、完整/部分主题标识

7. **legacy `.theme` 只保留兼容导入**
   - v1.6 后只导出 `.gmtheme`
   - 旧格式必须明确标记为“兼容导入/部分主题”

8. **5 套 shipped themes 必须同一完整度标准**
   - 不允许 1 套完整图片主题 + 4 套仅换色占位

> 文件名层面的实现（单 `theme.json` vs `manifest.json + theme.json`）可以让实现阶段决定；但**行为要求必须锁死**：元数据/预览/coverage 必须能在 apply 前读取，配置必须与资产 payload 清晰分层。

---

## 3. 架构边界（建议 roadmap 直接照此拆）

### Renderer / Editor 负责

- Theme Browser UI（内置 + 导入统一列表）
- 主题卡片、四视角预览、覆盖范围展示
- 用户动作编排：选择 / 预览 / 应用 / 导出入口
- 调用 script store 执行 `applyThemeBundle(...)`
- dirty 标记、保存、刷新资产列表

### Electron main 负责

- `.gmtheme` 文件 I/O
- ZIP 读取、schema 校验、路径安全校验
- 资产安装到项目 `assets/ui/themes/<themeId>/`
- 导入 remap / overwrite / skip 策略
- 导出时收集真实项目资产并重新打包

### Script/store boundary 负责

- 最终只把规范化后的 bundle config 写回现有 `ui.*` 结构
- 不负责文件复制
- apply 必须一次 push undo state

### 关键边界原则

1. **built-in 与 imported 只在“来源”不同，apply path 必须相同**
2. **preview、apply、export 必须共用同一 normalization 规则**
3. **项目一旦应用主题，就必须 project-local 自包含**

---

## 4. 主要风险（roadmap 必须正面处理）

### R1. 预览链路与实际应用链路分叉
- 后果：卡片好看，应用后不一致
- 对策：preview/apply 共用同一 bundle snapshot / normalization pipeline

### R2. 沿用现有 ad hoc 资产导入语义
- 后果：重复导入生成 `-1/-2/-3`，manifest 失配，垃圾文件失控
- 对策：主题包导入必须走命名空间 + hash/overwrite 规则，不能复用通用 uniqueFilename 逻辑

### R3. 过早并行生产 5 套主题
- 后果：contract 一改，返工 x5
- 对策：先冻结 package contract、安装链路、preview parity，再做 golden theme

### R4. 旧格式“看似兼容，实际静默降级”
- 后果：用户以为导入了完整主题，实际只换了颜色
- 对策：legacy import 必须显式标成部分主题，并展示缺失 coverage

### R5. built-in 资源不安装到项目
- 后果：编辑器能看，导出丢资源，项目不可搬运
- 对策：built-in 主题应用时也复制到项目内命名空间

---

## 5. 建议 phase order（给 roadmap 的直接答案）

### Phase 1 — Package contract 冻结
**目的：** 先锁格式、coverage、asset ownership、兼容策略。  
**产出：**
- `.gmtheme` v1.6 contract
- canonical path 规则
- namespace / overwrite / legacy import policy
- “完整主题” coverage checklist

### Phase 2 — Import / install / apply pipeline
**目的：** 跑通真正可安装的主题链路。  
**产出：**
- main-process install/export IPC
- 文件校验、引用 remap、原子 apply、undo
- built-in 与 imported 共用 apply path

### Phase 3 — Theme Browser / preview UX
**目的：** 让用户在 apply 前能看懂主题。  
**产出：**
- 图像卡片列表
- 四视角预览
- 完整/部分主题、coverage、来源、兼容信息展示
- 导入摘要 → 确认 → 应用三段式流程

### Phase 4 — Golden theme（第一套完整主题）
**目的：** 用 1 套主题验证工具链和验收标准。  
**建议主题：** 优先选 **和风古典** 或用户已最确定的“黄金样板主题”。  
**产出：**
- 完整素材清单
- 实际 `.gmtheme` 包
- apply/save/reopen/export 全链路 smoke test

### Phase 5 — 剩余 4 套主题批量产出
**目的：** 在 golden theme 验收模板上批量复制。  
**产出：**
- 其余 4 套完整图片主题
- 全部达到同一 coverage bar
- 卡片文案、标签、预览图集补齐

### Phase 6 — 收尾与生命周期打磨（可压缩）
**产出：**
- 重导/更新提示
- 已安装主题状态显示
- 删除主题/清理策略

---

## 6. 给 requirements 的一句话总判断

**v1.6 不该写成“支持主题包导入导出 + 5 套主题”。应该写成：先建立完整主题包契约与安装链路，再用 1 套 golden theme 验证，最后扩展到剩余 4 套 shipped themes。**

---

## 7. 规划时仍需注意的未决点

1. 包内是否采用 `theme.json` 单文件，还是 `manifest.json + theme.json` 分离  
   - 建议 requirement 锁“行为分层”，不要在此处卡 roadmap

2. 是否需要 project-local theme registry  
   - 若要支持“导入后下次打开项目仍可浏览/管理”，建议做轻量 registry

3. 5 套主题的最终题材命名  
   - 当前研究共识是：和风 / 现代 / 奇幻 / 浅色日常 / 高识别科幻（或同等差异化组合）

---

## 8. 最终建议

对 roadmap 的最强建议只有一条：

> **先做完整主题包系统，再做第一套 golden theme，最后再并行扩展剩余 4 套。**

如果反过来先堆 5 套素材，v1.6 最容易变成“看起来内容很多，但导入、预览、导出都不稳定”的半成品。
