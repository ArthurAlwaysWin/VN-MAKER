# v1.7 Architecture Research — 数据模型与持久化（Galgame 标配系统补全）

**Project:** VN-MAKER  
**Scope:** variable definitions, numeric ops, persistent unlocks, endings, CG gallery, project-vs-player data  
**Confidence:** HIGH (repo-based)

## Current state

- `script.json` is already the runtime truth source and should stay that way.
- Runtime is **page-based**, not old flat-command based; `docs/script-format.md` is partially stale.
- Save slots are file-based in `<project>/saves/`.
- Read history is separate in `localStorage` and keyed by **title** (`ReadHistory`), which is unstable.
- Config is also separate in `localStorage`.
- Biggest architectural risk today: **player-owned state is split across multiple stores with different keys**.

## Strong recommendations

1. **Keep all author-defined truth in `script.json`; keep all player state out of it.**  
   `script.json` should define variables, endings, gallery entries, unlockable IDs, labels, defaults, ordering.  
   Player progress should live only in runtime persistence.

2. **Add a stable immutable `projectId` / `gameId`.**  
   Do not key read history / unlocks by title anymore. Title rename must not erase progress.

3. **Introduce one runtime persistence facade:** `PlayerDataRepository`  
   Backends:
   - Electron/editor preview/exported game: `app.getPath('userData')/games/<projectId>/...`
   - Web: IndexedDB  
   Namespaces inside it:
   - `profile` = persistent cross-playthrough data
   - `saves` = slot snapshots + quicksave
   - optionally later `config`

4. **Do not make generic persistent variables the v1.7 MVP.**  
   Keep variables **run-scoped** first.  
   Persistent data should be dedicated, typed registries: read history, endings, CG unlocks.  
   This avoids a second hidden truth source and keeps debugging sane.

5. **Do not add a full expression language.**  
   Use a small normalized effect DSL reused everywhere.

## Recommended project-owned schema (`script.json`)

```json
{
  "systems": {
    "variables": {
      "affection_rin": {
        "type": "number",
        "label": "凛好感",
        "initial": 0,
        "min": 0,
        "max": 100
      },
      "route_rin": {
        "type": "boolean",
        "label": "凛线锁定",
        "initial": false
      }
    },
    "endings": {
      "rin_good": {
        "title": "凛 Good End",
        "order": 10,
        "category": "good",
        "cgIds": ["cg_rin_confession"]
      }
    },
    "gallery": {
      "cg": {
        "cg_rin_confession": {
          "title": "告白",
          "order": 10,
          "thumb": "ui/gallery/rin_confession_thumb.png",
          "images": [
            "cg/rin_confession_1.png",
            "cg/rin_confession_2.png"
          ]
        }
      }
    }
  }
}
```

### Rules

- Definitions live in `script.json`.
- Unlock state never lives in `script.json`.
- Gallery should be **explicit registry**, not inferred from assets folder.
- Endings and CGs should be referenced by **ID**, not raw image path.

## Recommended runtime effect model

Normalize all variable/unlock changes into one effect array:

```json
[
  { "type": "var:set", "id": "route_rin", "value": true },
  { "type": "var:add", "id": "affection_rin", "value": 5 },
  { "type": "var:sub", "id": "affection_rin", "value": 1 },
  { "type": "var:clamp", "id": "affection_rin", "min": 0, "max": 100 },
  { "type": "unlock:ending", "id": "rin_good" },
  { "type": "unlock:cg", "id": "cg_rin_confession" }
]
```

### Why this is the right brownfield move

- Reuses current `variables Map`.
- Replaces special-case `option.setVariable` with one extensible mechanism.
- Avoids expression parser complexity.
- Same effect runner can be used on page enter, choice select, ending pages later.

## Recommended player-owned persistence

### `profile.json` (cross-playthrough truth)

```json
{
  "version": 1,
  "projectId": "gm_xxxxx",
  "readHistory": {
    "pages": ["sceneA:0", "sceneA:1"]
  },
  "unlocks": {
    "endings": {
      "rin_good": {
        "unlockedAt": 1745800000000,
        "count": 1,
        "lastUnlockedAt": 1745800000000
      }
    },
    "cg": {
      "cg_rin_confession": {
        "unlockedAt": 1745800000000,
        "viewed": false
      }
    }
  }
}
```

### Save slot files

Keep them as **run snapshots only**:

```json
{
  "version": 2,
  "state": {
    "currentScene": "...",
    "pageIndex": 3,
    "dialogueIndex": 1,
    "variables": {},
    "history": [],
    "expressionState": {}
  }
}
```

### Separation rule

- **Save slot = transient run**
- **Profile = persistent progression**
- Save load must **not** be the authority for endings / CG unlocks / read history

That is the key design to avoid “two truth sources”.

## Runtime architecture split

- `ScriptDefinitionRegistry`
  - read-only view over `script.json.systems`
- `RunState`
  - current playthrough variables (`Map`)
- `PlayerProfile`
  - read history + endings + CG unlocks
- `PlayerDataRepository`
  - load/save profile + slots, backend-dependent

## Migration plan

1. Add stable `projectId`.
2. Create `PlayerDataRepository` abstraction.
3. Move `ReadHistory` into `profile`.
4. Migrate old `readHistory:<title>` into `profile` once.
5. Migrate `<project>/saves` to userData-based player storage.
6. Keep legacy read fallback for one release.
7. Keep reading old `option.setVariable`; normalize internally to `effects`.

## Strongest recommendations

- **Author data in `script.json`, player data in `profile/saves`, never mixed.**
- **Use stable `projectId`, not title, for all persistent keys.**
- **Ship a small effect DSL, not a scripting language.**
- **Make CG/endings explicit registries in `script.json`; store only unlocked IDs in player profile.**
- **Do not make save files the truth for persistent unlocks.**
