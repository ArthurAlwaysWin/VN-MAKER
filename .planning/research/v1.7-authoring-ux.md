# v1.7 Research Note — Authoring / Editor UX for「Galgame 标配系统补全」

**Project:** Galgame Maker  
**Focus:** no-code GUI authoring UX, first milestone slice  
**Date:** 2026-04-28  
**Confidence:** MEDIUM

## Current product reality

- 编辑器强项已经很清楚：**场景/页面树 + 画布 + 右侧 Inspector**，适合“看得见就能改”的 PPT 式创作。
- 运行时已经有基础逻辑地基：`variables`、`choice`、`condition` page、scene jump。
- 但作者侧 UX 还不成体系：变量没有项目级管理，条件页没有完整可视化入口，结局/CG 也没有“注册表 + 检查面板”。
- 因此 v1.7 不该做成“再加几个字段”，而应该做成一个**可被作者理解的轻量剧情系统工作台**。

## Strong recommendation

### 做一个新的顶层工作区：**「剧情系统」**

不要把变量、结局、CG 全塞进当前页面 Inspector。  
它们本质上都是**项目级资产/规则**，需要列表、搜索、状态、引用关系。

推荐结构：

1. **变量**
2. **结局**
3. **CG 图鉴**

而页面内只保留“消费这些定义”的入口：

- 选择页：给选项附加变量变化
- 条件页：从变量表里挑条件并选择跳转目标
- 某些页面/场景：标记“解锁 CG”或“进入结局”

这比“全靠右侧表单”更像产品，而不是 JSON 壳。

---

## 1) 变量定义与编辑：应该怎么露出

### 推荐模式：**注册表 + Inspector**

左侧列表：

- 变量名
- 类型：`布尔 / 数值`
- 分组：`剧情标记 / 好感度 / 收集 / 隐藏`
- 默认值
- 使用次数

右侧 Inspector：

- 显示名
- 内部键名（自动生成，可改）
- 默认值
- 描述
- 标签（如“好感度”）
- 引用位置列表

### 关键 UX 决策

- **默认只支持布尔和数值**。第一版不要上字符串/表达式变量。
- **好感度不是单独系统类型**，本质就是“带标签的数值变量”。这样最小且统一。
- 任何地方输入不存在的变量时，都提供 **“立即创建变量”**，避免来回跳页面。

### 为什么这样做

对 no-code 用户来说，变量最难的是“我到底有哪些变量、它们在哪里被用到”。  
先解决可见性和可管理性，比先做高级表达式更重要。

---

## 2) 条件分支 authoring：应该怎么做

### 推荐模式：**独立页面类型「条件页」**

不要做脚本式 if/else，也不要第一版就做复杂流程图。  
沿用当前 page-based 模型，直接加入可视化 **条件页** 最自然。

条件页建议字段：

- 条件组模式：`全部满足 / 任一满足`
- 条件行：
  - 变量
  - 运算符
  - 值
- 成功跳转场景
- 失败跳转场景

### 展示方式

在场景树中直接显示摘要：

`🔀 若 樱好感 >= 5 → 樱线入口，否则 → 日常线`

在 Inspector 中使用“自然语言”表单，而不是裸字段堆砌。

### 第一版范围建议

- 支持最多 1~3 条条件行
- 支持 `== != > >= < <=`
- 目标一律用**场景选择器**，不要让用户手写 scene ID

### 为什么这样做

这和现有 SceneTree / PageInspector 心智模型完全一致，学习成本最低。  
作者会把它理解成“这一页负责判定并分流”，而不是“我在写逻辑代码”。

---

## 3) 好感度 / 结局规则：第一版怎么做才小而完整

### 推荐模式：**先做“结局注册表 + 规则卡”，不要做通用规则引擎**

第一版不要开放任意复杂公式。  
建议只支持：

- 每个结局有：
  - 名称
  - 类型：`Normal / Bad / True / Hidden`
  - 进入场景
  - 解锁条件（条件组）
  - 优先级

### 好感度 authoring

在变量表里把某数值变量标记为“好感度”。  
在选择页选项卡上提供简化入口：

- `樱 +1`
- `凛 -1`
- `不变化`

而不是要求作者自己写 setVariable 对象。

### 结局判定 UX

推荐两种入口：

1. **项目级结局表**：统一看所有 ending
2. **结局场景标记**：在某个 scene 上勾选“这是一个结局入口”

并给出状态提示：

- 未被任何规则引用
- 有多个规则可能命中
- 没有默认兜底结局

### 为什么这样做

Galgame 作者真正要的不是“编程自由”，而是**我能不能清楚地做出路线和结局**。  
“结局注册表”比“开放表达式编辑器”更接近他们的任务。

---

## 4) CG 注册与解锁检查：第一版怎么做

### 推荐模式：**项目级 CG 图鉴管理器**

不要把 CG 当普通图片文件夹自己悟。  
要有明确的“CG 已注册 / 未注册”状态。

每个 CG 条目建议包含：

- 标题
- 关联图片资源
- 分组（角色 / 线路 / 通用）
- 缩略图
- 解锁来源：
  - 到达某 scene/page
  - 达成某 ending
- 备注

### 检查视图必须有

至少做两个视角：

1. **注册视角**：哪些图片已纳入图鉴
2. **解锁视角**：这个 CG 会被什么触发、是否有孤儿条目

最好再补一个轻量预览：

- 模拟状态下显示“已解锁 / 未解锁”
- 可从某 CG 反查到对应 scene / ending

### 第一版不要做

- 玩家端完整鉴赏室编辑器深度定制
- 多张图组成一个事件回放包
- 复杂分页/筛选系统

第一版先把 **“注册、触发、检查”** 闭环做出来。

---

## Smallest coherent product-grade slice

如果 v1.7 只做最小但完整的一刀，我建议是：

### A. 新增顶层「剧情系统」工作区

- **变量表**
- **结局表**
- **CG 图鉴表**

### B. 在页面编辑器补两个强入口

- **选择页**：选项里直接增减变量
- **条件页**：可视化条件 + 场景跳转

### C. 加一个轻量“规则检查/模拟”面板

至少能看：

- 某变量当前模拟值
- 某结局是否满足
- 某 CG 是否会被解锁

这三个合在一起，用户会第一次感觉：  
**“这个编辑器不只是排页面，也真的能做 Galgame 的路线、好感和收集系统。”**

---

## Explicit non-goals for v1.7

- 不做流程图编辑器
- 不做通用表达式语言
- 不做脚本模式
- 不做 BGM 图鉴
- 不做复杂成就系统

这些都比“先把变量/条件/结局/CG 变成可管理对象”更后。

---

## Implementation-shaping UX details

- 所有跳转都显示**场景名**，内部再存 ID
- 所有变量都显示**显示名 + 标签**，不要只显示 key
- 所有规则都给**自然语言摘要**
- 所有项目级对象都给**引用计数 / 反向引用**
- 所有新增动作都尽量支持**就地创建**（比如条件页里新建变量）
- 所有逻辑对象都要有**空状态引导**

## Final recommendation

**v1.7 不要理解为“给引擎补逻辑功能”，而要理解为“把隐藏逻辑翻译成 no-code 作者能掌控的 GUI 对象”。**

最优先的不是复杂度，而是：

1. 变量先可见
2. 条件先可编辑
3. 结局先可注册/检查
4. CG 先可注册/检查

只要这四件事形成一个统一的“剧情系统”工作区，哪怕规则能力还很基础，产品感就会明显成立。
