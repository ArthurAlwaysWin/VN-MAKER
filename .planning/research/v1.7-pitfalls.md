# v1.7 Pitfalls Research — Galgame 标配系统补全

**Scope:** 变量管理、好感度、结局追踪、CG 鉴赏  
**Date:** 2026-04-28  
**Confidence:** HIGH — based on direct repo inspection

## Executive Take

v1.7 最大风险不是“功能不会做”，而是把 **session state、save state、persistent unlock state、editor authoring state** 混成一团，最后造成旧项目可打开但不可信、旧存档可读取但逻辑漂移、编辑器看似支持但用户不会正确配置。

当前仓库已经有 **最薄的一层变量基础**：`ScriptEngine.variables`、choice `setVariable`、condition page runtime 判定、save/restore 持久化 variables。  
但它还 **没有**：项目级变量注册表、条件页完整 GUI、persistent unlock profile、ending/CG canonical ID contract、相关迁移与验证闭环。

## Strongest Risks

### 1) Save compatibility rewrite risk
- `engine.getState()` 当前只存：`currentScene/pageIndex/dialogueIndex/variables/history/expressionState`
- Electron save JSON 是 `version: 2`，Web save record 是 `version: 1`
- `restoreState()` 基本是“按字段直接吃”，没有正式 migration layer

**Implication:**  
新增 affection / ending / gallery 字段时，不能假设所有旧存档都有这些字段，更不能把“persistent unlock”塞进普通 slot state 里。

**Recommendation:**  
- 普通存档继续只存“当局状态”
- 新增单独 `persistent profile`（结局解锁、CG 解锁）
- `restoreState()` 必须 tolerant：缺字段按默认值补，不报错，不重置旧变量

### 2) 当前编辑器对“变量系统”其实还没闭环
- choice 页只有单个 `setVariable` 键值输入，且语义是 **增量相加**
- runtime 支持 `condition` page，但当前 PageEditor/SceneTree 没看到完整 condition 配置 UI；右键切换页型也只在普通页/选择页之间切
- 这意味着“引擎有一点，编辑器没有真正可教会用户的 authoring flow”

**Implication:**  
如果直接上“好感度系统”，用户会同时面对变量名命名、增减语义、条件页配置、结局判定，极易混乱。

**Recommendation:**  
- 先做 **变量注册表 + 类型/用途预设**（flag / number / affection）
- 再做 **最小运算集**：set / add / subtract
- 再做 **条件 builder**
- “好感度”只应是变量系统上的 preset，不要先单独做一套平行系统

### 3) Persistent unlock identity 设计错了会后面全返工
- 当前跨存档持久化只有 `ReadHistory`，而且 key 用的是 `readHistory:${projectId}`，`projectId` 实际来自 `script.meta.title`
- 标题可改名，不是稳定 ID
- 结局追踪 / CG 解锁如果也照这个思路做，会出现改标题后全部丢失或串档

**Recommendation:**  
- 不要用 title 当 persistent key
- 为项目引入稳定 `gameId/projectId`
- ending / cg 都必须先有 canonical ID，再谈 unlock

### 4) 结局追踪和 CG 鉴赏不能靠“运行时猜”
- 当前 end flow 只是结束后 2 秒回标题，没有 ending ID、ending metadata、unlock write point
- 当前也没有 CG registry / asset ownership contract

**Implication:**  
如果 v1.7 试图做“自动根据用过的背景/图片推断 CG”，大概率误判、漏判、导出断链、资产重命名后失效。

**Recommendation:**  
- 结局必须显式声明 `endingId`
- CG 必须显式注册到 gallery registry
- 解锁事件必须显式触发，不要做隐式资源扫描式“智能解锁”

### 5) Verification trap: 文档/旧界面存在漂移
- README 仍写“变量、条件跳转已完成”
- `docs/script-format.md` 仍有 commands 风格残留
- `src/editor/views/Scenes.vue` 仍是旧 command editor 痕迹
- 实际主线是 page-based editor/runtime

**Implication:**  
如果 v1.7 验证仍参考旧文档或旧视图，会误判“已有能力很多”，最后 roadmap 过度乐观。

**Recommendation:**  
- v1.7 只以 **page-based schema + current PageEditor + runtime save/load/export** 为准
- 先补一份单一权威 schema 文档，再做 milestone 验收

## Brownfield-Specific Warnings

### Migration risk
- 旧项目可能没有变量注册表、ending registry、cg registry、persistent profile
- 迁移策略应为 **lazy + additive**，不是打开项目就大改 schema

### UX confusion risk
- “设置变量”现在显示成 `变量名 = 值`，但 runtime 语义是 `current + value`
- 这对新手非常误导

**Must fix in v1.7:** 文案和 UI 明确区分：
- 设为固定值
- 增加/减少数值
- 条件判断

### Over-scope risk
最危险的诱惑是一次性打包：
- 变量系统
- 好感度模板
- ending list
- CG gallery
- BGM gallery
- 流程图
- achievement

这样会直接把 v1.7 做成半个平台。

## Recommended Phasing

### Phase 1 — Contract first
先冻结：
- variable registry schema
- operation contract（set/add/subtract only）
- condition contract
- stable project/game ID
- persistent profile schema
- ending registry / CG registry schema
- save compatibility rules

### Phase 2 — Variable system MVP
交付：
- 项目级变量管理
- choice/page 内可视化运算
- 条件页完整 GUI
- rename/delete reference scan

### Phase 3 — Affection + ending tracking
交付：
- affection preset variables
- ending IDs + ending unlock write path
- title/menu 内最小 ending list

### Phase 4 — CG gallery (only if contract is stable)
交付最小版：
- registry-backed gallery
- locked/unlocked state
- export/runtime parity

**If schedule slips:**  
优先保 `变量 + 好感度 + 结局追踪`，**把 CG 鉴赏 UI 延后**，不要反过来。

## What to Defer from v1.7

明确建议延期：
- 自动识别 CG（不要做）
- BGM 鉴赏
- 流程图
- 成就系统
- 复杂公式/脚本表达式
- 多条件组合编辑器（AND/OR 嵌套）
- gallery 缩略图生成/分页美化/筛选排序
- 任意资源都能当解锁物的通用插件化系统

## Validation Gates

v1.7 没有这些 gate 就不要算完成：

1. **Old project open gate**  
   v1.6 项目打开后无强制迁移失败、无字段丢失

2. **Old save load gate**  
   既有 slot / quicksave 可正常恢复，缺失新字段时走默认值

3. **Persistent separation gate**  
   解锁结局/CG 后，新开游戏仍保留；删除普通存档不影响 unlock

4. **Rename/delete safety gate**  
   重命名变量、endingId、CG 资源时要么批量更新引用，要么阻止操作

5. **Export parity gate**  
   Web 导出与桌面导出都能保留 ending/CG 解锁逻辑与资源引用

6. **Truth-source gate**  
   用 current PageEditor + runtime + save/load 验证，不用旧 command editor 文档当依据

## Bottom-Line Recommendation

**v1.7 应该先做“数据契约和可验证的变量/解锁基础设施”，再做标配玩法 UI。**  
最稳路线是：

1. 变量注册表 + 条件/运算 GUI  
2. 好感度 preset + ending tracking  
3. 最后才是 CG gallery

如果必须砍范围，**先砍 CG 鉴赏 UI，不要砍 save/persistent contract。**
