# v1.7 Research Note — Galgame 标配系统补全

**Project:** VN-MAKER  
**Focus:** gameplay systems / first milestone scope  
**Date:** 2026-04-28  
**Confidence:** MEDIUM

## Current fit

Repo inspection shows the engine already has:
- runtime `variables = new Map()` foundation
- `set_variable` + `condition` support
- save/load snapshotting of variables
- no real editor-side variable registry / picker / meta progression layer yet

So v1.7 should **build on the existing variable map**, not replace it.

## What mature VN engines/games do

### 1) Variables / flags
Mature engines treat this as a **registry**, not ad-hoc free-text keys:
- gameplay variables are referenced by stable IDs
- authors pick variables from UI, not manually type every key
- variable ops are minimal but complete: `set`, `add`, `subtract`, compare
- save-slot progression and meta-progression are usually separated

**Important pattern:**  
Ren'Py and Naninovel both distinguish **save-scoped state** from **persistent/global state**.  
That separation matters for endings / gallery unlocks.

### 2) Affection / favorability
In mature VNs, affection is usually **not a special engine subsystem**.  
It is usually just:
- an integer variable per route/character
- editor sugar for quick creation and adjustment
- threshold checks for route lock-in / ending branching

So “好感度系统” should be a **preset on top of variables**, not a separate runtime model.

### 3) Ending / unlock tracking
Mature engines handle endings as **explicit IDs in persistent data**:
- ending reached → write unlock record to meta storage
- extras / title menu can show locked vs unlocked
- often stores simple stats like first clear / clear count

This is better than inferring endings from save files.

### 4) CG gallery
Mature galleries are usually **registry-driven**:
- gallery item ID
- thumbnail / locked placeholder
- one item can contain one or multiple images
- unlock is explicit via condition or unlock call
- unlock state is persistent across saves

Do **not** auto-unlock by raw image filename usage.

## What SHOULD be in v1.7

### P0 — must ship in first milestone

#### A. Variable Registry + visual management UI
Add a project-level variable manager with:
- `id`
- display name
- type: `bool` / `int`
- default value
- group/category
- optional notes

And use variable pickers in commands instead of free-text names.

Minimum command support:
- `set`
- `add`
- `subtract`
- compare in condition
- choice option can apply variable effects

**Why:** this upgrades the current Map foundation into something authors can actually use safely.

#### B. Affection as a variable preset
Add “好感度变量” as a convenience template:
- underlying type = `int`
- optional linked character
- quick +/- buttons in editor
- recommended threshold checks

**Do not** build formula systems, decay, hidden UI widgets, or relationship graphs in v1.7.

#### C. Persistent ending tracking
Add an explicit ending registry:
- ending ID
- display name
- locked title / hidden title mode
- optional description
- optional thumbnail

At runtime:
- `end` should optionally carry `endingId`
- reaching it writes to persistent meta storage
- extras/title can show unlocked state

**Minimum persistence data:** unlocked yes/no, first reached timestamp, clear count.

#### D. CG gallery with explicit unlocks
Add a project-level CG registry:
- CG item ID
- display name
- thumbnail
- locked thumbnail/placeholder
- one-to-many images per item

At runtime:
- explicit `unlock_cg(id)` or equivalent
- persistent unlock state
- gallery screen shows locked slots before unlock
- click unlocked item → simple image viewer

**Recommended model:** unlock by **gallery item/group ID**, not by asset path.

## What to DEFER

Do not put these in v1.7 first milestone:
- flowchart / branch graph
- replay scene system
- BGM room
- generic global-variable editor for all use cases
- string variables / expression language expansion
- achievement system
- auto-detect CG unlock from “image appeared once”
- complex affection formulas / decay / route planner UI
- Steam sync / cloud meta progression

## Strongest product recommendations

1. **Make affection a preset, not a separate engine system.**  
   This keeps scope small and matches mature VN practice.

2. **Separate save variables from persistent unlock data.**  
   Do not mix route stats with ending/CG meta progress.

3. **Use explicit registries for endings and CGs.**  
   IDs first, assets second.

4. **Ship the editor UX, not just runtime support.**  
   For this milestone, the real value is authoring usability: variable manager, pickers, quick affection setup, ending/CG registries.

## Suggested v1.7 cut line

### Include
- variable manager
- variable picker in commands
- int/bool vars
- set/add/subtract/compare
- affection preset
- ending registry + persistent unlock
- CG registry + persistent unlock
- simple extras screens for endings + CG gallery

### Defer
- replay / BGM / flowchart / advanced formulas / achievements
