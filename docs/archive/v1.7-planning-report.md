# v1.7 Planning Report — Galgame 标配系统补全

**Date:** 2026-04-28  
**Status:** Planned  
**Milestone:** v1.7 Galgame 标配系统补全

## 1. 为什么是这个 milestone

v1.6 已经把主题包、浏览器、安装/应用/导出链路和 5 套 shipped 完整主题做到了较完整状态。  
接下来最明显的产品短板，不再是“视觉包装不够”，而是 **Galgame 标配的剧情系统能力还不完整**：

- 变量 / flag 还没有产品级 authoring UX
- 好感度只是概念，没有正式工作流
- 结局追踪 / CG 鉴赏还没有 persistent progression 闭环

因此 v1.7 的目标不是继续补视觉，而是补齐 **作者能配置、运行时可信、跨周目不串档** 的剧情系统基础设施。

## 2. 最终确定的 scope

### In scope

1. **变量注册表**
   - 项目级变量定义
   - bool / number 类型
   - 显示名、默认值、分组、备注

2. **页面内变量效果与条件 GUI**
   - `set` / `add` / `subtract`
   - 1~3 条条件组合
   - `== != > >= < <=`
   - 场景选择器 + 自然语言摘要

3. **好感度预设**
   - 本质仍是数值变量
   - 编辑器提供角色标签和 `+1/-1` 糖层

4. **结局注册表 + persistent unlock**
   - ending ID、标题、分类、顺序
   - 解锁状态、首次达成时间、达成次数

5. **CG 注册表 + 显式解锁 + 最小图鉴**
   - CG item ID、缩略图、锁定占位图、图片列表
   - locked/unlocked 展示
   - 简单查看器

6. **save/profile 分层与验证护栏**
   - save slot = 当前跑团
   - profile = 跨周目 progression
   - clean contract + 明确 reset / versioning 语义
   - orphan / missing reference / 无兜底结局等验证

### Deferred

- 分支流程图
- replay scene system
- BGM room
- achievements
- string variables / expression language
- generic persistent variables
- auto-detected unlocks
- cloud / Steam sync

## 3. 关键产品决策

### 决策 A：**变量系统是主轴**

v1.7 不做成“好感度系统 + 结局系统 + 图鉴系统”三套平行机制。  
统一原则是：**变量系统是基础，其他能力都建立在它上面。**

**结果：**
- 好感度不是单独 runtime 子系统
- 条件判断不需要第二套规则引擎
- 作者心智模型更稳定

### 决策 B：**好感度只是变量预设**

好感度的本质是带标签的 number 变量。  
本 milestone 只给它更友好的 authoring UX，不单独发明复杂公式、关系图或衰减系统。

**结果：**
- scope 更可控
- runtime 复杂度更低
- 后续如果要增强，也能在变量系统上平滑扩展

### 决策 C：**结局与 CG 必须显式注册、显式解锁**

不允许靠图片出现过、场景走到哪、存档里有什么去“猜”解锁。  
一切都必须是显式 ID + 显式 unlock effect。

**结果：**
- 数据更稳定
- 重命名 / 移动资源不容易炸
- 验证和反向引用扫描才有可能做可靠

### 决策 D：**save slot 与 persistent profile 必须分层**

普通存档只表示当前跑团快照；  
结局、CG、read-history 这类跨周目进度必须独立进入 profile。

**结果：**
- 删除存档不会清空收集进度
- 旧存档兼容策略更清晰
- 导出成品与编辑器预览更容易保持一致

## 4. 关键技术选型

### 4.1 Author data vs Player data 分离

**作者数据** 放在 `script.json`：
- variables
- endings
- gallery.cg

**玩家数据** 放在运行时持久化：
- profile
- saves

这是本 milestone 最重要的技术边界。

### 4.2 引入稳定 `projectId`

当前 read-history 这类数据不能继续依赖标题名等可变字段。  
v1.7 需要稳定 `projectId` / `gameId` 来绑定 persistent progression。

### 4.3 使用最小 effect DSL

推荐统一为：

- `var:set`
- `var:add`
- `var:sub`
- `unlock:ending`
- `unlock:cg`

而不是继续在 choice / page / ending 各处加互不兼容的特例字段。

### 4.4 Registry-first schema

endings 和 CG 都先注册，再被引用，再被解锁。  
这样 authoring、runtime、validation、export 才能共享同一套 ID 真相源。

### 4.5 不为“旧项目迁移”牺牲 v1.7 contract

当前项目还没有真正上线，因此 v1.7 不把“旧项目 / 旧存档迁移兼容”当成阻塞项。  
技术选择优先级改为：

1. schema 清晰
2. profile/save 分层清晰
3. reset / rebuild 语义明确
4. 为未来真正上线后的 versioning 预留空间

## 5. 风险与 cut line

### 最大风险

1. 把 save state 和 persistent unlock state 混在一起
2. 继续用 title 当持久化 key
3. 先做 runtime、后补 authoring，导致功能能跑但作者不会用
4. scope 膨胀到流程图 / BGM / 成就 / 复杂公式

### Cut line

如果 scope 开始失控，优先级必须是：

1. **保住变量系统注册化**
2. **保住 ending/profile contract**
3. **保住兼容 / 持久化 / validation 护栏**
4. **CG 图鉴 UI 可以后移，但 CG registry / unlock contract 不应该砍**

## 7. 下一步建议

推荐直接从 **Phase 83** 开始：

- 先做 contract / profile / compatibility
- 再做 GUI authoring
- 最后再做 ending / gallery UI

这样可以避免把 v1.7 做成“界面先很好看，但底层 contract 返工”的路线。
