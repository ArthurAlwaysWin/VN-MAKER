## 优先级建议

按"对玩家体验影响 x 实现难度"排序：

### P0 — Galgame 标配，缺了像半成品

1. **CG 鉴赏室 + 结局追踪** — 通关后回看 CG 是 Galgame 最核心的二周目动力。需要：CG 注册表、解锁追踪（persistent storage）、鉴赏界面
2. **好感度/数值系统** — 编辑器里可视化管理变量，引擎支持数值运算和条件判断。已有 variables Map 基础，主要缺编辑器 UI 和数值运算指令
3. **立绘动画（至少基础的）** — 抖动、跳跃、淡入滑入。不需要 ATL 那么强，但静态立绘缺乏生命力。CSS animation/transition 就能实现

### P1 — 明显提升体验

4. **镜头效果** — 震屏、画面缩放。container transform 就能做，成本不高但效果显著
5. **分支流程图** — 可视化显示场景间的连接关系。这是"PPT 式编辑"理念的自然延伸
6. **更多转场效果** — 至少补上 dissolve、wipe、zoom，现在只有 3 种太少
7. **富文本/内联标记** — 文字颜色、大小、等待、抖动。类似 Ren'Py 的 `{color=#f00}红字{/color}`

### P2 — 锦上添花

8. 视频播放
9. 移动端导出（Capacitor/TWA）
10. BGM 鉴赏室
11. 粒子/天气特效
12. 多语言支持

## 六、UI 设计系统深度对比

> 基于对全部 UI 代码（style.css 1284 行、7 个 UI 组件类、ThemeManager、tokens.js、widgetDefaults.js）的逐行审查。

### 本质区别：图片驱动 vs CSS 驱动

**成熟引擎的做法（Ren'Py / TyranoBuilder / KiraKira）**：

UI 的视觉主体是**美术素材**。对话框是画师画的 PNG，按钮是 normal/hover/pressed 三态图片，菜单背景是精心设计的插画，存档卡片边框是装饰性素材。引擎做的是"把这些图片放到正确的位置"。

Ren'Py 典型做法：
```python
# 对话框 = 画好的图，Frame() 九宫格拉伸
style say_window:
    background Frame("gui/textbox.png", 30, 30, 30, 30)

# 按钮 = 三张图，不是 CSS 颜色变化
style button:
    background "gui/button/idle.png"
    hover_background "gui/button/hover.png"
    selected_background "gui/button/selected.png"
```

TyranoBuilder 同理 — 对话框、菜单、按钮全部支持替换为自定义图片。制作者买一套"UI 素材包"，替换进去，游戏外观就完全不同了。

**本项目当前做法**：

UI 视觉主体是 **CSS 属性**。对话框是 `background: linear-gradient(...)` + `backdrop-filter: blur()` + `border-radius`。按钮是 `background: rgba(...)` 的颜色变化。整体是"现代毛玻璃"风格的 CSS UI。

### 逐元素差距对比

| UI 元素 | 成熟引擎 | 本项目现状 | 差距 |
|---------|---------|-----------|------|
| **对话框** | 画师画的装饰性 PNG，边角有花纹/纹理 | CSS gradient + blur + border-radius | 看起来像"软件界面"而非"游戏界面" |
| **按钮** | 三态图片（idle/hover/pressed），可以是任意形状 | CSS 颜色变化，矩形 | 缺乏视觉个性 |
| **面板/窗口** | 整张背景图或装饰性边框图 | CSS rgba 半透明 + blur | 同上 |
| **九宫格** | 标配 | ✅ 已有 applyNineSlice，6 个目标元素 | **方向正确，但覆盖面不够** |
| **装饰元素** | 花纹边角、分割线图片、光效图层 | ❌ 无 | "精致感"的关键来源 |
| **自定义光标** | 主题化鼠标指针 | ❌ 无 | 小细节大感觉 |
| **按钮图标** | 风格化图片图标 | SVG/emoji + 文字 | 缺乏统一美术风格 |
| **全屏背景图** | 菜单/存读档有独立背景插画 | 仅 GameMenu 有 backgroundImage | 其他界面缺失 |

### 已有基础（做对了的部分）

1. **九宫格系统** — `ThemeManager.applyNineSlice()` 已支持 dialogueBox、menuPanel、saveSlot、choiceButton、titleButton、settingsPanel
2. **widgetStyles backgroundImage** — panel 和 button 都有背景图字段
3. **slider trackImage/thumbImage** — 滑块图片替换
4. **标题页完全图片驱动** — TitleScreen 支持自定义背景 + 任意图片/文字/按钮元素

### 核心缺口

**1. 对话框 — 最关键的 UI 元素**

九宫格可以套上，但缺少更丰富的图片装饰支持：
- 没有名牌独立背景图（很多 VN 的名牌是一个装饰条 PNG）
- 没有对话框装饰层（底部花纹、角落装饰）
- 九宫格是 `::before` 伪元素单层，无法叠加装饰

**2. 按钮 — 全场景缺少图片态**

- GameMenu 按钮有 `icon` 但没有按钮整体背景图三态
- SaveLoad 界面按钮、分页 tab 全部纯 CSS
- 九宫格三态仅 choiceButton 和 titleButton 接入，其他按钮（game-menu-button、save-load-close、qab-btn 等）未接入

**3. 全屏界面背景 — 大面积缺失**

- SaveLoadScreen：无背景图配置（仅 panel-bg CSS token）
- BacklogScreen：同上
- SettingsScreen：有 settings-bg-layer 但不完整
- 成熟引擎这些界面通常有精心设计的背景插画

**4. 装饰层 — 完全空白**

成熟 VN 大量使用装饰性图片元素 — 花纹边角、光效图层、分割线。不是功能性的，纯粹为了"精致感"。本项目没有这个概念。

### 问题本质

主题系统在**参数化配色**上已经很强（OKLCH 智能配色、41 token、控件形状），但在**图片资产作为 UI 视觉主体**这条路上还没走通。目前 UI 无论怎么调都像"一个漂亮的软件设置界面"，而不是"一个有风格的游戏界面"。

差距不是"少了某个功能"，而是**视觉表达通道不够宽**：CSS 配色参数是窄路，图片素材才是宽路。需要让制作者能用一套美术素材"皮肤"整个游戏 UI。

## 设计目标：三层自定义体系

> 以下是本项目 UI/游戏设计的长期目标框架。

### 第一层：开箱即用（零学习成本）

**一键完全配置。** 内置 5 套完整的 UI 系统（包含全部图片素材），用户一键应用到自己的游戏项目。

- 每套主题包含：配色 token + 九宫格图片 + 按钮三态图 + 界面背景 + 装饰素材 + 控件风格 + 布局参数
- 覆盖全部游戏界面：标题页、对话框、游戏菜单、存读档、回想、设置页
- 风格差异明显（如：现代简约、日式传统、奇幻、赛博朋克、复古学院）
- 用户体验：选一个主题 → 整个游戏 UI 焕然一新，不需要理解任何参数

**现状**: 已有 5 套内置主题（default/wafuu/modern-sky/fantasy-dark/minimal-white），但仅覆盖配色 + 控件形状 + 部分布局，缺少图片素材层。

### 第二层：有限自定义（低学习成本）

**不碰代码，最大自由度。** 通过编辑器 GUI 让用户在预设框架内自由调整。

- 替换 UI 图片素材（对话框背景图、按钮图、界面背景）
- 调整配色（当前 OKLCH 智能配色 + token 系统已覆盖）
- 调整布局参数（当前 widgetStyles + screenLayout 已覆盖）
- 混搭不同主题的不同部分（如：用 A 主题的对话框 + B 主题的菜单）
- 用户体验：需要一些学习成本理解"哪个参数控制什么"，但始终在 GUI 内操作

**现状**: 配色和布局参数编辑已较完善，图片素材替换通路尚未打通。

### 第三层：完全自由（高级用户）

**为有能力和意愿的用户提供极高自由度。** 不仅限于 UI，也包括更复杂的游戏逻辑。

- 自定义脚本/逻辑（变量运算、条件分支、自定义命令）
- 自定义 UI 组件（超越预设元素类型，自行编排界面）
- 插件/扩展系统
- 导出后二次开发
- 目标：实现与 Ren'Py、TyranoBuilder 同等的功能上限

**现状**: 尚未开始。这是长期愿景。

## 总结

本项目的编辑器体验和主题系统已经**超过大部分竞品**，但在**游戏演出力**（动画、镜头、特效）和**Galgame 系统功能**（CG 回收、好感度、结局追踪）两个维度与成熟引擎有明显差距。**UI 设计系统**方面，参数化配色很强但图片资产驱动的视觉表达通路尚未打通，是当前最影响"游戏感"的短板。

三层自定义体系（开箱即用 → 有限自定义 → 完全自由）是长期设计目标，按"引擎能力 → 编辑器配套 → 内容产出"的顺序推进。
