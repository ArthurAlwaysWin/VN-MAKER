## 六、UI 设计系统深度对比

> 基于对全部 UI 代码（style.css 1284 行、7 个 UI 组件类、ThemeManager、tokens.js、widgetDefaults.js）的逐行审查。

### 本质区别：图片驱动 vs CSS 驱动

**成熟引擎的做法（Ren'Py / TyranoBuilder / KiraKira）**：

UI 的视觉主体是**美术素材**。对话框是画师画的 PNG，按钮是 normal/hover/pressed 三态图片，菜单背景是精心设计的插画，存档卡片边框是装饰性素材。引擎做的是"把这些图片放到正确的位置"。

Ren'Py 典型做法：
```python
# 对话框 = 画好的图，Frame() 九宫格拉伸
style say_window:
    background Frame("gui/textbox.png", 30, 30, 30, 30)

# 按钮 = 三张图，不是 CSS 颜色变化
style button:
    background "gui/button/idle.png"
    hover_background "gui/button/hover.png"
    selected_background "gui/button/selected.png"
```

TyranoBuilder 同理 — 对话框、菜单、按钮全部支持替换为自定义图片。制作者买一套"UI 素材包"，替换进去，游戏外观就完全不同了。

**本项目当前做法（截至 v1.6）**：

UI 现在是 **图片资产 + CSS 参数混合驱动**。对话框、按钮族、major screens、cursor、action icon、title/title-inclusive full theme 都已经支持图片化与项目内资产落盘；CSS token / 布局参数继续负责可调节性与 fallback 外观。

### 逐元素差距对比

| UI 元素 | 成熟引擎 | 本项目现状 | 差距 |
|---------|---------|-----------|------|
| **对话框** | 画师画的装饰性 PNG，边角有花纹/纹理 | ✅ 名牌背景 / 装饰层 / 图片化主题通路已完成（v1.5） | 已补齐主要通路 |
| **按钮** | 三态图片（idle/hover/pressed），可以是任意形状 | ✅ 多按钮族图片态 + 主题图标（v1.5） | 已补齐主要通路 |
| **面板/窗口** | 整张背景图或装饰性边框图 | ✅ major screens 背景 / chrome / 装饰层 + shipped theme assets（v1.5/v1.6） | 已补齐主要通路 |
| **九宫格** | 标配 | ✅ 已有 applyNineSlice + 图片 UI 全链路 | 方向已验证 |
| **装饰元素** | 花纹边角、分割线图片、光效图层 | ⚠️ 已支持装饰层，但素材生态仍早期 | 从“无”到“可用” |
| **自定义光标** | 主题化鼠标指针 | ✅ 已支持主题光标（v1.5） | 已完成 |
| **按钮图标** | 风格化图片图标 | ✅ 主题图片图标 / fallback 闭环（v1.5） | 已完成 |
| **全屏背景图** | 菜单/存读档有独立背景插画 | ✅ SaveLoad / Backlog / GameMenu / Settings 全部已支持（v1.5） | 已完成 |

### 已有基础（做对了的部分）

1. **九宫格系统** — `ThemeManager.applyNineSlice()` 已支持 dialogueBox、menuPanel、saveSlot、choiceButton、titleButton、settingsPanel
2. **widgetStyles backgroundImage** — panel 和 button 都有背景图字段
3. **slider trackImage/thumbImage** — 滑块图片替换
4. **标题页完全图片驱动** — TitleScreen 支持自定义背景 + 任意图片/文字/按钮元素

### 核心缺口（更新后）

原始 v1.3 视角里的 **对话框 / 按钮 / 全屏界面背景 / cursor / icon** 图片通路，已在 **v1.5-v1.6** 主要补齐。  
从当前状态看，UI 设计系统剩下更值得关注的不是“能不能接图片”，而是：

1. **更细粒度的主题覆盖** — 例如每场景 / 每页面主题切换，而不只是项目级完整主题。
2. **更完整的素材生态** — 例如字体资源、更多 shipped presets、社区主题分发，而不只是 5 套内置完整主题。
3. **更高级的未应用预览与混搭能力** — 当前 browser 已有静态预览与 coverage 说明，但 live preview / 部分混搭仍是后续空间。

### 问题本质

主题系统在**参数化配色**上已经很强（OKLCH 智能配色、41 token、控件形状），而“图片资产作为 UI 视觉主体”的关键通路已在 v1.5-v1.6 主要打通。当前更现实的差距，已经从“能不能图片化”转向“素材生态够不够深、覆盖粒度够不够细”。

差距不是"少了某个功能"，而是**视觉表达通道不够宽**：CSS 配色参数是窄路，图片素材才是宽路。需要让制作者能用一套美术素材"皮肤"整个游戏 UI。

## 设计目标：三层自定义体系

> 以下是本项目 UI/游戏设计的长期目标框架。

### 第一层：开箱即用（零学习成本）

**一键完全配置。** 内置 5 套完整的 UI 系统（包含全部图片素材），用户一键应用到自己的游戏项目。

- 每套主题包含：配色 token + 九宫格图片 + 按钮三态图 + 界面背景 + 装饰素材 + 控件风格 + 布局参数
- 覆盖全部游戏界面：标题页、对话框、游戏菜单、存读档、回想、设置页
- 风格差异明显（如：现代简约、日式传统、奇幻、赛博朋克、复古学院）
- 用户体验：选一个主题 → 整个游戏 UI 焕然一新，不需要理解任何参数

**现状（已更新至 v1.6）**: 已有 5 套 shipped 完整主题（default / wafuu / modern-sky / fantasy-dark / minimal-white），并且已经通过 shared install/apply/export/browser pipeline 闭环；缺口从“没有完整主题”转向“更多主题生态与更细粒度覆盖”。

### 第二层：有限自定义（低学习成本）

**不碰代码，最大自由度。** 通过编辑器 GUI 让用户在预设框架内自由调整。

- 替换 UI 图片素材（对话框背景图、按钮图、界面背景）
- 调整配色（当前 OKLCH 智能配色 + token 系统已覆盖）
- 调整布局参数（当前 widgetStyles + screenLayout 已覆盖）
- 混搭不同主题的不同部分（如：用 A 主题的对话框 + B 主题的菜单）
- 用户体验：需要一些学习成本理解"哪个参数控制什么"，但始终在 GUI 内操作

**现状（已更新至 v1.6）**: 配色、布局参数、图片素材替换通路都已打通；下一阶段更像是做 per-scene/per-page override、字体资源、混搭和更高级预览。

### 第三层：完全自由（高级用户）

**为有能力和意愿的用户提供极高自由度。** 不仅限于 UI，也包括更复杂的游戏逻辑。

- 自定义脚本/逻辑（变量运算、条件分支、自定义命令）
- 自定义 UI 组件（超越预设元素类型，自行编排界面）
- 插件/扩展系统
- 导出后二次开发
- 目标：实现与 Ren'Py、TyranoBuilder 同等的功能上限

**现状**: 尚未开始。这是长期愿景。

## 总结

本项目的编辑器体验和主题系统已经**超过大部分竞品**，但在**Galgame 系统功能**（CG 回收、好感度、结局追踪）和更高阶演出（粒子/天气、富文本、视频）两个维度与成熟引擎仍有明显差距。**UI 设计系统**方面，原先“图片资产驱动通路尚未打通”的短板已在 v1.5-v1.6 主要补齐，当前更像是在补生态深度与高级能力。

三层自定义体系（开箱即用 → 有限自定义 → 完全自由）是长期设计目标，按"引擎能力 → 编辑器配套 → 内容产出"的顺序推进。
