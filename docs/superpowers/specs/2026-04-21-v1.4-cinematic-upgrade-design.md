# v1.4 Design Spec — 演出力升级

## Summary

v1.4 的目标是补齐当前最影响“游戏感”的短板：让角色、镜头、转场从“能用”提升到“有演出感”，同时保持项目一贯的低门槛、可视化、预设化路线。

本版本不追求做成 Ren'Py ATL 级自由动画系统，而是交付一套稳定、可预览、可导出的基础演出工具箱：

1. 立绘预设动画
2. 镜头效果
3. 编辑器配置与预览
4. 更多页面转场

## Why Now

当前项目最强的是：

- PPT 式可视化编辑
- 实时预览
- 主题系统与参数化 UI

当前最明显的短板是：

- 立绘缺少动态表现
- 镜头语言基本为空
- 转场选项过少

这会让成品“像一个可配置的软件界面”，而不是“像一款真正的视觉小说”。v1.4 应优先解决这个问题，因为它对用户第一印象、成品质感、截图观感和试玩体验的提升最直接。

## Product Goal

让制作者在**不写代码**的前提下，通过编辑器就能做出更有动感、更像商业 VN 的页面演出。

成功标准不是自由度极高，而是：

- 学习成本低
- 预览稳定
- 导出一致
- 结果明显可感知

## Scope

### In Scope

#### Phase 61 — 立绘预设动画（引擎）

为角色提供 6-8 种预设动画，例如：

- shake
- bounce
- slide-in
- fade-in
- nod
- breathe

CharacterLayer 负责播放、替换和停止这些动画。

#### Phase 62 — 镜头效果（引擎）

为舞台容器提供一组高性价比镜头效果：

- shake
- zoom
- pan
- flash

镜头效果作为页面级事件触发，不属于单个角色。

#### Phase 63 — 编辑器动画/镜头配置

在编辑器中补齐配置入口和预览：

- PageInspector 角色行增加动画下拉框
- 页面新增镜头效果配置入口
- iframe 运行时预览支持演出效果回放
- 画布模式仅负责静态配置编辑与可视占位提示，不承担演出正确性预览

#### Phase 64 — 更多转场效果

在现有 `fade / slide / none` 基础上扩展：

- dissolve
- wipe
- scale
- blur

复用现有页面切换与 BackgroundLayer 机制，不开第二套转场系统。

### Out of Scope

以下内容不纳入 v1.4：

- 自由动画脚本语言
- ATL / 时间轴 / 曲线编辑器
- CG 鉴赏 / 结局追踪 / 好感度系统
- UI 图片驱动体系
- `.gmtheme` 图片资产打包升级
- 粒子系统
- 视频播放
- 插件系统

## Design Principles

### 1. 预设优先，不开放底层自由编排

v1.4 只允许用户选择动画名和有限参数，不暴露底层 keyframes、缓动曲线或组合表达式。

### 2. 一页就是一个镜头

页面仍然是核心叙事单位。动画、镜头、转场都围绕“页面进入和页面切换”发生，避免引入持续运行的复杂状态机。

### 3. 引擎能力先于编辑器配套

先让引擎稳定支持预设效果，再在编辑器里开放配置和预览，避免 UI 先行导致数据和实现返工。

### 4. 预览与导出一致

编辑器中的预览效果必须尽量与 Web / 桌面导出保持一致；如果某效果在预览端不稳定，应优先降级而非硬上。

### 5. 不依赖新增美术资源

v1.4 的所有效果基于 CSS、transform、opacity、overlay 等现有能力实现，不依赖新图片素材。

## Architecture

### Character Animation

#### Data Model

角色实例增加可选动画字段，例如：

```json
{
  "characterId": "heroine-a",
  "expression": "smile",
  "animation": "breathe"
}
```

动画字段是命名预设，不是参数化表达式。

v1.4 不开放自定义动画参数。角色侧最终合同为：

- `animation: string | undefined`
- `undefined` 或 `none` 表示不播放动画
- 仅允许预设注册表中的名称

首版支持的最小预设集应至少包含以下 6 种，推荐交付 7 种：

- `fade-in`
- `slide-in-left`
- `slide-in-right`
- `shake`
- `nod`
- `breathe`
- `bounce`（推荐）

#### Runtime Responsibility

`CharacterLayer` 负责：

- 将动画名映射到 CSS class / keyframes
- 在角色进入页面或更新页面时触发动画
- 管理替换、停止和重复触发
- 保证快速翻页时不会残留旧动画状态

#### Lifecycle

- 页面激活后，角色 DOM 完成挂载，再触发角色动画
- one-shot 动画播放结束后自动清理 class
- loop 动画（如 `breathe`）持续到页面离开、角色被替换或用户跳到下一页
- 页面离开时必须清理全部角色动画 class 和临时 inline style

### Camera Effects

#### Data Model

新增独立页面级 `camera` 配置，挂在页面对象本身，而不是角色对象内部。首版合同固定为单个可选对象：

```json
{
  "camera": {
    "effect": "shake",
    "durationMs": 300,
    "intensity": "medium",
    "direction": "horizontal",
    "trigger": "onEnter"
  }
}
```

字段约束：

- `effect`: `shake | zoom | pan | flash`
- `durationMs`: number，取值范围 `100-2000`，超出时在编辑器侧 clamp 到边界值
- `intensity`: `low | medium | high`
- `direction`: 仅在 `pan` 或定向 `shake` 时启用
- `trigger`: 首版固定为 `onEnter`

`direction` 的允许值固定为：

- `shake`: `horizontal | vertical | both`
- `pan`: `left | right | up | down`
- `zoom`: 不使用 `direction`
- `flash`: 不使用 `direction`

首版不支持：

- 同一页面多个 camera 效果串联
- 条件触发
- 表达式计算
- 时间轴

#### Runtime Responsibility

舞台容器统一负责镜头效果：

- shake：容器短时位移
- zoom：容器缩放
- pan：容器平移
- flash：覆盖层闪白或闪黑

镜头逻辑必须集中在单一舞台层，不允许分散到 DialogueBox、CharacterLayer、BackgroundLayer 内部各自处理。

#### Lifecycle

- 页面切换完成、新页面成为活动页后，若存在 `camera` 配置，则立即触发一次
- 同一时刻只允许一个 camera 效果活跃
- 新页面触发新镜头时，旧镜头必须先清理
- 页面离开时必须清理舞台容器上的 transform / filter / overlay 状态

### Page Transitions

页面转场继续沿用现有机制，只扩展转场类型枚举与实现逻辑。目标是把它当作“现有能力增强”，而不是独立子系统。

为避免与 camera `zoom` 语义冲突，转场侧不使用 `zoom` 这一命名，统一使用 `scale` 表示页面切换时的缩放型过渡。

v1.4 明确要求转场与角色动画、镜头效果按以下顺序衔接：

1. 清理旧页面残留的角色动画与镜头状态
2. 执行页面切换转场
3. 新页面激活并挂载 DOM
4. 触发角色动画
5. 触发页面级 `camera`

这样可避免三类效果同时争夺同一层 transform。

## Editor Design

### Character Animation Configuration

在 `PageInspector` 的角色配置区域增加动画选择控件：

- 默认：none
- 选项：预设动画列表
- 仅展示当前版本已支持的稳定预设

### Camera Effect Configuration

在页面级配置区域增加镜头效果入口：

- 效果类型
- 持续时间
- 有限强度或方向参数（仅在确有必要时开放）

### Preview

v1.4 的**唯一权威预览路径**是现有 iframe 运行时预览链路。

原因：

- 动画、镜头、转场都依赖真实运行时层级与生命周期
- 画布模式更适合静态编辑，不适合作为演出效果的准确定义来源

因此本版本约束为：

- `PageInspector` 中与演出相关的“预览/重播”操作统一驱动 iframe 运行时预览
- 画布模式可保留静态配置展示，但不是演出正确性的判定依据
- 若 iframe 预览上下文未准备好，编辑器必须显示明确原因，而不是静默失败

编辑器预览应满足：

- 改动后能立即重播当前效果
- 当前条件不满足时给出明确提示
- 不引入第二套“仅编辑器可见”的演出逻辑

#### Preview UI Contract

编辑器中所有演出预览都通过明确的用户操作触发，不做隐式后台播放：

- 角色动画：动画下拉框旁提供“重播”按钮
- 镜头效果：镜头配置区域提供“预览效果”按钮
- 页面转场：在页面切换相关控件旁提供“预览转场”按钮

iframe 预览未就绪时：

- 按钮显示 disabled 状态
- UI 显示明确原因，如“预览未启动”或“当前页面无法重播该效果”
- 不做静默失败
- 不自动退回画布模式执行假预览

## Error Handling

### Unknown Preset

若读到未知动画名或未知镜头效果：

- 运行时忽略该效果
- 在开发态输出明确警告
- 不允许因单个错误配置导致整页崩溃

### Effect Conflicts

同一时刻只允许一个全局镜头效果生效：

- 后触发覆盖前一个
- 不做多效果叠加调度
- 避免容器 transform 失控

### Fast Navigation

快速翻页、跳过、自动播放时：

- 旧页面的演出效果必须被及时清理
- 新页面的效果必须可预测地重新开始
- 不允许残留 class、transform 或 overlay 状态

## Integration Notes

### Existing Systems to Reuse

- 页面数据模型与页面切换机制
- `CharacterLayer`
- `BackgroundLayer`
- 编辑器现有 `PageInspector`
- 现有画布 / iframe 预览链路

### Existing Systems Not to Expand in v1.4

- 不引入新的脚本语言
- 不扩展成事件时间轴系统
- 不把动画逻辑塞进主题系统

## Phase Dependency Boundaries

v1.4 是一个**单里程碑、多 phase**的交付，不是一个单实现块。四个 phase 的依赖关系明确如下：

1. **Phase 61（角色动画）**
   - 提供角色侧预设动画运行时基础
   - 不依赖其他 v1.4 phase
2. **Phase 62（镜头效果）**
   - 提供舞台级 camera 运行时基础
   - 不依赖 Phase 61
3. **Phase 63（编辑器配置与预览）**
   - 依赖 Phase 61 与 Phase 62 的最终合同
   - 只消费运行时能力，不定义新运行时语义
4. **Phase 64（更多转场）**
   - 依赖现有页面切换机制
   - 需要与 61/62 明确层级衔接，但实现上可独立推进

因此在执行计划上：

- 61 和 62 可以并行研究，但运行时合同应先稳定
- 63 必须在 61/62 合同冻结后开始
- 64 可与 63 并行收尾，但需要共享同一套“切页清理顺序”定义

## Testing and Verification Focus

v1.4 需要重点验证：

1. 角色动画在普通播放、自动模式、快速翻页下表现稳定
2. 镜头效果不会破坏 UI 层布局和交互
3. 转场新增后不会回归现有页面切换逻辑
4. 编辑器预览与导出结果基本一致
5. 未知配置和冲突配置可被安全降级

## Milestone Shape

建议按以下节奏推进：

1. 先做引擎侧预设动画
2. 再做镜头效果
3. 再接入编辑器配置与预览
4. 最后扩展转场效果并统一回归

原因是：动画和镜头属于基础能力，编辑器是它们的消费方；先打地基，再补 UI，返工最少。

## Risks

### Risk 1 — 过度追求自由度

如果过早引入复杂参数、组合动画或脚本表达式，会迅速把 v1.4 从“高性价比升级”拖成“新平台工程”。

**Mitigation:** 严格限制为预设化能力。

### Risk 2 — 编辑器预览与运行时分叉

如果预览走一套实现、运行时走另一套实现，后续维护成本会快速上升。

**Mitigation:** 预览尽量复用真实运行链路，只做必要壳层适配。

### Risk 3 — transform 冲突

角色动画、镜头效果、页面转场都可能使用 transform，容易互相覆盖。

**Mitigation:** 提前明确层级边界：

- 角色动画作用于角色层
- 镜头效果作用于舞台容器
- 转场作用于页面切换层

## Acceptance Criteria

- 编辑器提供角色动画下拉框，且至少可选择 6 种预设动画
- 角色动画可在 iframe 预览中重播，并在导出后的 Web / 桌面运行时生效
- 页面对象支持单个可选 `camera` 配置，且 `shake / zoom / pan / flash` 四类效果均可在预览与运行时触发
- 页面转场总数从现有 3 种扩展到至少 7 种：`none / fade / slide / dissolve / wipe / scale / blur`
- 未知动画名、未知 camera 效果、非法枚举值均会安全 no-op，并在开发态给出警告
- 页面切换后不会残留上一页的角色动画 class、舞台 transform 或 flash overlay 状态
- 现有不含新字段的旧项目可继续打开、预览、导出，不需要迁移脚本
- 用户无需写代码即可使用全部 v1.4 新能力

## Follow-on Work

v1.4 完成后，后续顺序建议保持为：

1. v1.5 — UI 图片驱动体系
2. v1.6 — 完整主题包与图片资产导入导出
3. v1.7 — 变量 / 好感度 / 分支流程图

原因是：

- v1.4 解决“动不动”
- v1.5 / v1.6 解决“像不像游戏”
- v1.7 再解决“系统够不够完整”
