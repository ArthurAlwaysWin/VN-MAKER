# v1.7 Phase 83 Report — 剧情系统契约与持久化护栏

**Status:** Complete  
**Phase:** 83  
**Requirements closed:** DATA-01, DATA-02, DATA-03, PERS-01, PERS-02

## What shipped

Phase 83 turned v1.7's gameplay foundation from ad-hoc runtime behavior into an explicit, testable contract:

- `script.json` now carries stable `projectId`, `contractVersion`, and explicit `systems.variables` / `systems.endings` / `systems.gallery.cg`
- player progression now lives behind `PlayerDataRepository`, separated from normal save slots
- runtime/editor writes now converge on canonical `effects[]`
- reset/rebuild flows now have named entrypoints instead of manual file edits

## Key technical decisions

### 1. Stable identity lives in author data

`projectId` is seeded additively in a shared contract helper and preserved on reload.  
This makes persistent progress independent from mutable title text.

### 2. Author data and player data are different authorities

`script.json` owns registries and definitions.  
`player-data/profile.json` owns cross-run truth like read history and unlocks.  
Save slots keep run snapshots only.

### 3. Writes converge on one minimal DSL

Phase 83 locked the write-side contract to:

- `var:set`
- `var:add`
- `var:sub`
- `unlock:ending`
- `unlock:cg`

Legacy `setVariable` remains read-compatible, but canonical saved data normalizes back to `effects[]`.

### 4. Reset is explicit, not accidental

Reset/rebuild is now surfaced as named operations, which keeps testing and iteration fast without depending on migration-heavy legacy behavior.

## Review follow-up fixed before closure

Two production-readiness issues were fixed before closing the phase:

- async read-history persistence failures are now caught on the real async path
- legacy variable compat edits no longer drop hidden `effects[]` entries during editor round-trip

## Why this phase matters

Without Phase 83, Phase 84-86 would have built authoring UI on unstable persistence rules.  
Now variables, endings, and CG systems can be added on top of one stable identity model, one persistence boundary, and one write contract.
